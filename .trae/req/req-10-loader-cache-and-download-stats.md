# 需求 10：C loader 缓存与下载统计修复

## 背景

用户反馈两个问题：
1. 每次构建都重新下载依赖、重新编译 C loader，耗时且重复劳动。依赖（wheel）缓存已实现，但 C loader 每次都重新编译。
2. 下载依赖阶段实际执行了下载，但汇总表格"下载"列显示 `-`，统计失效。

## 需求清单

- [x] 1. C loader 缓存：编译产物缓存到 `~/.fspack/cache/loaders/`，命中时直接复制到 dist，未命中才编译并回写缓存。
- [x] 2. 下载统计修复：`download_wheels` 在 pip download 前后统计 wheelhouse 目录 wheel 总大小差值，回写 `stage.add_bytes()`，使汇总表"下载"列显示实际字节数。
- [x] 3. 缓存命中时通过 `stage.hit_cache()` 记录，detail 标注"缓存命中"或编译器名。
- [x] 4. 新增/更新测试覆盖缓存命中与未命中路径、下载字节数回写。
- [x] 5. 门禁通过：ruff check/format、pyrefly、pytest --cov≥95%（实际 98.27%）。

## 不在范围内

- 不修改 wheel 缓存机制（已实现）。
- 不修改 embed/standalone 缓存（已实现）。
- 不严格区分"网络下载字节"与"缓存复用字节"——下载统计口径为"wheelhouse 目录本次新增 wheel 总大小"，包含从 `--find-links` 复制的本地缓存 wheel。该数字反映"本次构建处理的依赖体积"，对用户有参考价值。

## 验收标准

1. 重复构建同一项目时，C loader 阶段显示"缓存命中"，耗时显著下降。✓（469ms → 33ms）
2. 下载依赖阶段汇总表"下载"列显示实际字节数（如 `133.2MB`），不再为 `-`。✓
3. 全套门禁通过，覆盖率不低于当前值。✓（98.27%，loader.py 100%）
